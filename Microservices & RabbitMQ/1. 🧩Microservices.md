## 📌 Microservices Architecture

### **Definition**

Microservices architecture is an approach to software development where an application is structured as a collection of **small, loosely coupled, independently deployable services**.  
Each service focuses on a **single business capability** and communicates with other services through well-defined APIs, typically over HTTP or messaging systems.

---

## 🏗 **Core Principles**

- **Single Responsibility** → Each service does one thing well.    
- **Loose Coupling** → Services interact only via APIs, not shared databases.    
- **Independent Deployment** → Services can be deployed without affecting others.    
- **Decentralized Data Management** → Each service has its own database.    
- **Automation** → CI/CD pipelines, containerization, and orchestration.    

---

## 📂 **Microservices vs Monolith**

|Feature|Monolith|Microservices|
|---|---|---|
|Deployment|Single unit|Independent per service|
|Scaling|Whole application|Per service|
|Database|Shared|Independent per service|
|Codebase|Single|Multiple|
|Communication|In-process|Network-based (HTTP, gRPC, MQ)|

---

## 🔌 **Communication Between Services**

### **1. Synchronous Communication**

- REST (HTTP)    
- gRPC    
- GraphQL (less common in pure microservices)    

### **2. Asynchronous Communication**

- Message Brokers (RabbitMQ, Kafka, Azure Service Bus)    
- Event Streaming    
- Pub/Sub pattern    

**Example**:

```
sequenceDiagram
    OrderService->>PaymentService: POST /process-payment
    PaymentService-->>OrderService: Payment Confirmation
```

### 📊 Difference Between Streaming and Normal Requests

|Normal Request/Response|Streaming|
|---|---|
|One request → One response|Multiple messages over time|
|Wait for full process before sending response|Response can start immediately|
|Good for simple operations|Good for large data or real-time updates|

---

## 🛠 **Key Components**

### **1. Service**
A unit that implements a specific business function.
### **2. API Gateway**
Acts as a single entry point for clients, routing requests to appropriate services.
### **3. Database Per Service**
Ensures isolation and prevents coupling.
### **4. Service Registry**
Keeps track of active service instances (e.g., Eureka, Consul).
### **5. Message Broker**
Handles async communication (e.g., RabbitMQ, Kafka).

## ⚙ **Design Patterns in Microservices**

- **API Gateway Pattern**    
- **Database per Service**    
- **Saga Pattern** (for distributed transactions)    
- **CQRS** (Command Query Responsibility Segregation)    
- **Event Sourcing**    
- **Strangler Fig** (migrating from monolith to microservices)

## 📊 **Advantages**

- Independent scaling    
- Easier to adopt new technologies    
- Faster deployments    
- Better fault isolation    

---

## ⚠ **Challenges**

- **Complexity** in deployment and communication    
- **Data consistency** across services    
- **Network latency**    
- **Monitoring** distributed systems    

---

## 📌 **Best Practices**

- Use **Domain-Driven Design (DDD)** to define service boundaries    
- Keep services small but not too granular    
- Use **asynchronous messaging** for better scalability    
- Implement **centralized logging & monitoring**    
- Secure APIs using **OAuth2, JWT**
