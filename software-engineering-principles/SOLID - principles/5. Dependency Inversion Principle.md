## 🧩 Dependency Inversion Principle (DIP)

### 📖 Definition

> **High-level modules should not depend on low-level modules. Both should depend on abstractions.**  
> **Abstractions should not depend on details. Details should depend on abstractions.**

---

### 🚩 **Why DIP?**

- Reduces coupling between high-level logic and low-level implementations.    
- Makes systems flexible and easier to extend.    
- Enables swapping implementations (e.g., Cash → Credit → PayPal) without modifying high-level code.    

---

### 🛑 **Violation of DIP**

In this example, the `Order` class directly depends on concrete payment classes:
```
public class CashPayment
{
    public void Pay(decimal amount) 
        => Console.WriteLine($"Paid {amount} in cash.");
}

public class CreditCardPayment
{
    public void Pay(string cardNumber, decimal amount) 
        => Console.WriteLine($"Paid {amount} using credit card {cardNumber}.");
}

public class Order
{
    private readonly CashPayment _cashPayment = new CashPayment();

    public void Checkout(decimal amount)
    {
        _cashPayment.Pay(amount); // Directly tied to CashPayment!
    }
}
```
**Problems:**

- `Order` is **hardcoded to CashPayment**.    
- Switching to `CreditCardPayment` requires editing `Order` (violates OCP too).    
- No abstraction layer → tightly coupled.
### ✅ **Applying DIP**

We introduce an abstraction (`IPayment`) and make both `Order` and payments depend on it.

#### 1️⃣ Create the abstraction:
```
public interface IPayment
{
    void Pay(decimal amount);
}
```
2️⃣ Implement payment methods:
```
public class CashPayment : IPayment
{
    public void Pay(decimal amount)
        => Console.WriteLine($"Paid {amount} in cash.");
}

public class CreditCardPayment : IPayment
{
    public void Pay(decimal amount)
        => Console.WriteLine($"Paid {amount} using credit card.");
}
```
3️⃣ Inject the abstraction into `Order`:
```
public class Order
{
    private readonly IPayment _payment;

    public Order(IPayment payment)
    {
        _payment = payment; // Inject abstraction, not concrete class
    }

    public void Checkout(decimal amount)
    {
        _payment.Pay(amount);
    }
}
```
4️⃣ Usage:
```
var cashOrder = new Order(new CashPayment());
cashOrder.Checkout(100); // Paid 100 in cash.

var creditOrder = new Order(new CreditCardPayment());
creditOrder.Checkout(200); // Paid 200 using credit card.
```

Now:

- `Order` **depends on `IPayment` (abstraction)**, not a concrete class.    
- Adding `PayPalPayment` or `CryptoPayment` requires **no changes to `Order`**.    

---

### 💡 Key Takeaways

- **High-level modules (Order)** should not depend on **low-level details (Cash, Credit)**.    
- Use **interfaces or abstract classes** to invert dependencies.    
- Combine **DIP with Dependency Injection (DI)** for loose coupling.    

---

### 🔑 Real-World Examples

- **Logging:** High-level code depends on `ILogger`, not `FileLogger` or `DatabaseLogger`.    
- **Payment Processing:** `Order` depends on `IPayment`, not `Cash` or `Credit`.    
- **Repositories:** Controllers depend on `IRepository`, not `SqlRepository`.
