## 🧩 Interface Segregation Principle (ISP)

### 📖 Definition

> **No client should be forced to depend on methods it does not use.**

This means:

- Interfaces should be **small and specific**.    
- Avoid "fat" interfaces that force implementing classes to define unused methods.    

---

### 🚩 **Why ISP?**

- Prevents classes from having unnecessary methods.    
- Makes interfaces more maintainable and easier to implement.    
- Reduces the risk of **empty or meaningless implementations**.    

---

### 🛑 **Violation of ISP: Fat Interface**

Imagine a single interface for all workers:

```
public interface IWorker
{
    void Work();
    void Eat();
}

public class HumanWorker : IWorker
{
    public void Work() => Console.WriteLine("Human is working.");
    public void Eat() => Console.WriteLine("Human is eating lunch.");
}

public class RobotWorker : IWorker
{
    public void Work() => Console.WriteLine("Robot is working.");
    public void Eat() => throw new NotImplementedException("Robot does not eat!");
}
```
**Problem:**

- `RobotWorker` is forced to implement `Eat()`, even though it doesn't eat.    
- Violates **ISP** because `RobotWorker` depends on a method it doesn't use.
### ✅ **Applying ISP**

We split the interface into smaller, **role-specific interfaces**:
```
public interface IWorkable
{
    void Work();
}

public interface IFeedable
{
    void Eat();
}

public class HumanWorker : IWorkable, IFeedable
{
    public void Work() => Console.WriteLine("Human is working.");
    public void Eat() => Console.WriteLine("Human is eating lunch.");
}

public class RobotWorker : IWorkable
{
    public void Work() => Console.WriteLine("Robot is working.");
}
```

Now:

- `HumanWorker` implements both `IWorkable` and `IFeedable`.    
- `RobotWorker` only implements `IWorkable`.

### 🛑 **Violation of ISP: Fat Payment Interface**

A single large interface for all payment types:
```
public interface IPayment
{
    void PayWithCash(decimal amount);
    void PayWithCreditCard(string cardNumber, decimal amount);
    void PayWithPayPal(string email, decimal amount);
}

public class CashPayment : IPayment
{
    public void PayWithCash(decimal amount) => Console.WriteLine($"Paid {amount} in cash.");
    public void PayWithCreditCard(string cardNumber, decimal amount) 
        => throw new NotImplementedException("Cash payment does not use credit cards!");
    public void PayWithPayPal(string email, decimal amount)
        => throw new NotImplementedException("Cash payment does not use PayPal!");
}

public class CreditCardPayment : IPayment
{
    public void PayWithCash(decimal amount)
        => throw new NotImplementedException("Credit card payment does not use cash!");
    public void PayWithCreditCard(string cardNumber, decimal amount) 
        => Console.WriteLine($"Paid {amount} using credit card {cardNumber}.");
    public void PayWithPayPal(string email, decimal amount)
        => throw new NotImplementedException("Credit card payment does not use PayPal!");
}
```

**Problems:**

- Each payment class is forced to implement **unused methods**.    
- Leads to **empty implementations** or `NotImplementedException`.    
- Violates **ISP**.    

---

### ✅ **Applying ISP: Split Interfaces**

We break the fat interface into **smaller, focused interfaces**.
```
public interface ICashPayment
{
    void PayWithCash(decimal amount);
}

public interface ICreditCardPayment
{
    void PayWithCreditCard(string cardNumber, decimal amount);
}

public interface IPayPalPayment
{
    void PayWithPayPal(string email, decimal amount);
}
```
Now, implement only what is needed:
```
public class CashPayment : ICashPayment
{
    public void PayWithCash(decimal amount) 
        => Console.WriteLine($"Paid {amount} in cash.");
}

public class CreditCardPayment : ICreditCardPayment
{
    public void PayWithCreditCard(string cardNumber, decimal amount) 
        => Console.WriteLine($"Paid {amount} using credit card {cardNumber}.");
}

public class PayPalPayment : IPayPalPayment
{
    public void PayWithPayPal(string email, decimal amount) 
        => Console.WriteLine($"Paid {amount} using PayPal account {email}.");
}
```

Finally, the `Order` class depends only on the payment interface it needs:
```
public class Order
{
    public void Checkout(ICashPayment payment, decimal amount)
    {
        payment.PayWithCash(amount);
    }
}
```
Or use different methods for different payment types:
```
public class Order
{
    public void Checkout(ICreditCardPayment payment, string card, decimal amount)
        => payment.PayWithCreditCard(card, amount);

    public void Checkout(IPayPalPayment payment, string email, decimal amount)
        => payment.PayWithPayPal(email, amount);
}
```

---

### 🔑 Real-World Examples

- **Printers:** Instead of a single `IPrinter` with `Print`, `Scan`, `Fax`, split into `IPrinter`, `IScanner`, `IFax`.    
- **Payment Methods:** Separate interfaces for `ICreditCardPayment`, `IPayPalPayment`, `IBankTransfer`.    
- **Workers (as above):** Split `Work` and `Eat`.    

---

### 💡 Key Takeaways

- Break large interfaces into **smaller, focused interfaces**.    
- Classes should **only implement what they need**.    
- Helps keep systems **flexible and easier to maintain**.
