## 📌 Single Responsibility Principle (SRP)

### 🧠 **Definition**

> **A class should have only one reason to change.**

This means that **a class should only have one job or responsibility**. If a class has more than one responsibility, it becomes tightly coupled, making it harder to modify, extend, and test.

SRP is the **"S" in SOLID principles** of object-oriented design.

---

### ❌ **What happens if SRP is violated?**

- **Tight coupling**: Changes in one responsibility may unintentionally affect another.    
- **Low maintainability**: More responsibilities mean more complexity.    
- **Harder testing**: Testing becomes harder because multiple concerns are mixed.    
- **Poor scalability**: Adding new features can break unrelated parts.    

---

### ✅ **Why follow SRP?**

- **Better maintainability**: Each class focuses on a single concern.    
- **Easier to test**: Smaller, well-defined classes are easier to test in isolation.    
- **Improved reusability**: Classes with one purpose can be reused in different contexts.    
- **Simplifies debugging**: Easier to trace issues because each class has a clear scope.    

---

### 🔧 **Example (Violation of SRP)**

Here’s a class that violates SRP because it handles **two responsibilities**:

1. Managing employee data    
2. Saving employee data to a database    

```
public class Employee
{
    public string Name { get; set; }
    public string Position { get; set; }

    // Responsibility 1: Business logic
    public decimal CalculateSalary()
    {
        // Salary calculation logic
        return 5000;
    }

    // Responsibility 2: Data persistence
    public void SaveToDatabase()
    {
        // Code to save employee details to database
        Console.WriteLine("Employee saved to database.");
    }
}

```
**Problem:**

- If the database logic changes, we must modify the `Employee` class (even though it’s not related to employee logic).    
- This violates SRP.    

---

### ✅ **Correct Implementation (SRP Applied)**

We split responsibilities into **two classes**:

```
// Class 1: Employee handles business logic
public class Employee
{
    public string Name { get; set; }
    public string Position { get; set; }

    public decimal CalculateSalary()
    {
        // Salary calculation logic
        return 5000;
    }
}

// Class 2: Responsible for data persistence
public class EmployeeRepository
{
    public void Save(Employee employee)
    {
        // Code to save employee details to database
        Console.WriteLine($"Employee {employee.Name} saved to database.");
    }
}
```

Now:

- `Employee` handles only **business logic**.    
- `EmployeeRepository` handles **data persistence**.    

---

### 🏗 **Real-World Analogy**

Imagine a **chef** in a restaurant:

- The chef's job is to **cook food**, not **manage billing** or **clean tables**.    
- If the chef is also responsible for billing and cleaning, their main job suffers.    

Similarly, in SRP:
- **One class = one job**.    

---

### 🔑 **Key Points**

- Each class should have **only one reason to change**.    
- Delegate unrelated tasks to other classes.    
- Improves **readability**, **testability**, and **maintainability**.
