## 📖 What is Bearer Authentication?

**Bearer Authentication** (also known as **Token Authentication**) is an HTTP authentication scheme where the client sends an **access token** (a string) in the `Authorization` header to prove identity.

- The term **"Bearer"** means: whoever "bears" the token is granted access.    
- Commonly used with **OAuth 2.0** and **JWT (JSON Web Tokens)**.    
- Unlike Basic Auth, it doesn't send username/password in every request.

### 📦 Header format:

```
`Authorization: Bearer <token>`
```

---

## 🧠 How Does Bearer Authentication Work?

1. ✅ **Login:**
    
    - The user provides username/password to the **authentication server** (usually via `/login` endpoint).        
    - If valid, the server issues an **access token** (and sometimes a **refresh token**).        
    
2. 🚀 **Accessing protected resources:**
    
    - The client includes the **access token** in the `Authorization` header:
        
```
        `Authorization: Bearer eyJhbGciOiJIUzI1NiIsIn...`
```
        
    - The server verifies the token, extracts user info (claims), and allows or denies access.
        
3. 🔄 **Token Expiry and Refresh:**
    
    - Access tokens are usually short-lived (e.g., 15 minutes).        
    - When the access token expires, the client sends a **refresh token** to get a new access token.

🔄 Sequence Diagram
```
sequenceDiagram
    participant Client

    Client->>Server: POST /login (credentials)
    Server-->>Client: 200 OK (access_token)

```
## 🔄 Access Token vs Refresh Token

| Type              | Description                                                                            |
| ----------------- | -------------------------------------------------------------------------------------- |
| **Access Token**  | A short-lived token (e.g., 15 minutes) used to access protected resources.             |
| **Refresh Token** | A long-lived token (e.g., days/weeks) used to get a new access token without re-login. |
### 🧭 Example Flow

```
1. Client → POST /login → sends username + password
2. Server → returns:
   {
     access_token: "...",
     refresh_token: "..."
   }

3. Client → uses access_token to call API endpoints

4. Access token expires → client uses refresh_token to get new access token
```

---

### 📦 Sample JWT (Access Token):

```
{
  "sub": "1234567890",
  "name": "John Doe",
  "role": "Admin",
  "exp": 1717000000
}

```

- This payload is **digitally signed** by the server, so it can’t be tampered with.    
- The `exp` field is the expiration time.
