# 🐇 Observer Project Documentation

---

## 📚 Overview

This project demonstrates an **event-driven architecture** using **RabbitMQ** and **MassTransit** in .NET.

- Uses **MassTransit** as a high-level abstraction to simplify RabbitMQ integration.
- Implements **publish-subscribe messaging** pattern.
- Decouples producers and consumers using **message contracts** (events).
- Demonstrates payment workflow reacting to new orders.

---

## 🔑 Key Concepts

| Term                | Description                                  |
|---------------------|----------------------------------------------|
| RabbitMQ            | Open-source message broker implementing AMQP protocol. |
| MassTransit         | .NET framework simplifying message-based applications. |
| Publisher           | Sends events/messages to RabbitMQ exchanges. |
| Consumer            | Listens to queues to process messages.       |
| Event / Message      | Immutable data object passed between services. |
| Exchange            | Routes messages to queues based on routing keys. |
| Queue               | Stores messages until consumers process them. |

## 💡 Core Classes

### PaymentConsumer.cs

```
public class PaymentConsumer : IConsumer<OrderCreated>
{
    private readonly IPaymentPublisher _paymentPublisher;

    public PaymentConsumer(IPaymentPublisher paymentPublisher)
    {
        _paymentPublisher = paymentPublisher;
    }

    public async Task Consume(ConsumeContext<OrderCreated> context)
    {
        var order = context.Message;
        Console.WriteLine($"[PaymentConsumer] Processing payment for Order {order.OrderId} - Amount: {order.Amount}");

        // Simulate payment success or failure
        bool paymentSuccess = true; // For demo, always success

        if (paymentSuccess)
        {
            await _paymentPublisher.PublishPaymentSucceeded(order.OrderId, DateTime.UtcNow);
            Console.WriteLine($"[PaymentConsumer] Payment succeeded for Order {order.OrderId}");
        }
        else
        {
            await _paymentPublisher.PublishPaymentFailed(order.OrderId, "Insufficient funds");
            Console.WriteLine($"[PaymentConsumer] Payment failed for Order {order.OrderId}");
        }
    }
}
```

- Listens to `OrderCreated` events.    
- Calls payment logic (currently simulated).   
- Publishes payment result events.

## 🔧 Configuration - `AddRabbitMq` Extension Method

```
public static class ObserverExtensions
{
    public static IServiceCollection AddRabbitMq(this IServiceCollection services, string rabbitMqUri, string userName, string password)
    {
        services.AddMassTransit(x =>
        {
            x.AddConsumer<PaymentConsumer>();

            x.UsingRabbitMq((context, cfg) =>
            {
                cfg.Host(rabbitMqUri, h =>
                {
                    h.Username(userName);
                    h.Password(password);
                });

                cfg.ReceiveEndpoint("payment_queue", e =>
                {
                    e.ConfigureConsumer<PaymentConsumer>(context);
                });
            });
        });

        services.AddScoped<IOrderPublisher, OrderPublisher>();
        services.AddScoped<IPaymentPublisher, PaymentPublisher>();

        return services;
    }
}
```

- Registers MassTransit and the consumer.    
- Configures RabbitMQ connection.    
- Sets up the receive endpoint with the queue `payment_queue`.    
- Registers publishers for dependency injection.

# RabbitMQ In The Microservices 
## 🏗 **Architecture Overview**

We will build **2 microservices**:
1. **PalMazad Service** – Handles Orders, Users, Products, and Notifications.    
2. **Payment Service** – Dedicated to processing payments.    

They will communicate asynchronously via **RabbitMQ** using **events** (publish/subscribe pattern) with **MassTransit**.

---

## 🔄 **Event Flow**

1. **Create Order (PalMazad API)**    
    - User sends an API request to create an order.        
    - PalMazad stores the order in its database with status **Pending**.        
    - PalMazad publishes an **`OrderCreated`** event to RabbitMQ.        
2. **Payment Service consumes `OrderCreated**    
    - Listens to `OrderCreated` events.        
    - Simulates or processes actual payment.        
    - If successful → Publishes **`PaymentSucceeded`**.        
    - If failed → Publishes **`PaymentFailed`**.        
3. **PalMazad consumes `PaymentSucceeded**    
    - Updates the order status to **Paid** in its DB.        
    - Publishes an **`OrderPaid`** event.        
4. **Notification (inside PalMazad)**    
    - Listens to `OrderPaid` events.        
    - Sends email confirmation to the buyer.        
    - Optionally sends push notifications to the seller.

```
            Amount = order.Amount,

```

### Email Sending Process - Summary Documentation

#### Overview

The email sending process is integrated into the system as part of the **business logic layer** and triggered via **event-driven communication** between microservices using RabbitMQ and MassTransit.

##### Key Components:

- **EmailEvent**: An event representing the need to send an email.    
- **EmailHandler**: A MediatR notification handler that consumes the `EmailEvent`.    
- **EmailManager**: Business service responsible for actual email sending logic.    
- **MediatR**: Used for in-process communication (decoupling business logic and event handlers).    
- **MassTransit**: Used for inter-service event communication via RabbitMQ.    

---

#### Flow

1. **Event Publishing (In other microservices):**  
    When an action requires sending an email (e.g., payment succeeded), an `EmailEvent` is published via RabbitMQ.    
2. **Event Consumption:**  
    The **Observer** project consumes the `EmailEvent` from RabbitMQ.    
3. **Mediator Notification:**  
    Instead of directly calling the email sending service or using a MassTransit publisher inside the Observer, the event is passed to the **MediatR** mediator.    
4. **EmailHandler Executes:**  
    The `EmailHandler`, registered as an `INotificationHandler<EmailEvent>`, receives the event from MediatR and calls the **EmailManager** service.    
5. **EmailManager Sends Email:**  
    The EmailManager handles the SMTP or third-party email API integration to send the email.    

---

#### Why Use _mediator Instead of MassTransit Publisher for Email Sending?

- **Separation of Concerns:**  
    MassTransit is designed for **inter-service** asynchronous messaging via RabbitMQ, while MediatR is optimized for **in-process** communication. Email sending is part of the internal business logic after receiving an external event.
    
- **In-Process Decoupling:**  
    Using MediatR inside the same microservice (Observer or Business) decouples the event handling from business logic with minimal overhead and without the latency or complexity of an external bus call.
    
- **Avoiding Unnecessary Message Passing:**  
    Publishing email send requests again to the message broker (MassTransit) would add overhead and complexity when all needed logic lives in the same process boundary.
    
- **Simplified Dependency Management:**  
    Mediator facilitates easier injection and testability within the same service without adding cross-service message dependencies.
