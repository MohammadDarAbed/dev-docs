# 🧭 Messaging in .NET with MediatR and MassTransit (RabbitMQ)

## 📌 Overview

In distributed systems or microservices, we often need to communicate between components or services using **events** or **commands**.

In .NET, you can use:

- **`MediatR`** for **in-process (local)** communication.
    
- **`MassTransit` + `IPublishEndpoint`** for **cross-service (distributed)** communication via **RabbitMQ**.
    

---

## 🧱 Components Summary

|Tool|Type|Scope|Durable|Use Case|
|---|---|---|---|---|
|`IMediator.Publish`|In-process pub/sub|Same service|❌ No|Triggering local events between components|
|`IPublishEndpoint.Publish`|Distributed messaging|Cross services|✅ Yes|Publishing events to other services via RabbitMQ|

---

## 🧠 When to Use What

### ✅ Use `IMediator` when:

- The event/command is handled **within the same service**    
- You want to **decouple business logic**    
- Durability is **not** critical (loss is acceptable in failure)    

### ✅ Use `IPublishEndpoint` when:

- The event needs to reach **other services**    
- You want **durable delivery** (even if the app crashes)    
- You want to **retry** or use a **dead-letter queue**    

---

## 📬 `IMediator.Publish` (Local Event)

### Description

- Publishes an in-process `INotification` to all registered `INotificationHandler<T>` handlers.

### Example

```
await _mediator.Publish(new EmailEvent
{
    To = "user@example.com",
    Subject = "Hello",
    Body = "This is an in-process event."
});
```
### EmailEvent.cs

```
public record EmailEvent : INotification
{
    public string To { get; init; }
    public string Subject { get; init; }
    public string Body { get; init; }
}
```
### EmailEventHandler.cs

```
public class EmailEventHandler : INotificationHandler<EmailEvent>
{
    public Task Handle(EmailEvent notification, CancellationToken cancellationToken)
    {
        Console.WriteLine($"Sending email to {notification.To}");
        return Task.CompletedTask;
    }
}
```
---

## 📦 `IPublishEndpoint.Publish` (RabbitMQ Message)

### Description

- Publishes a message to RabbitMQ through MassTransit.    
- This is for **cross-service communication**.    

### Usage in a Consumer or Service

```
public class PaymentSucceededConsumer : IConsumer<PaymentSucceeded>
{
    private readonly IPublishEndpoint _publishEndpoint;

    public PaymentSucceededConsumer(IPublishEndpoint publishEndpoint)
    {
        _publishEndpoint = publishEndpoint;
    }

    public async Task Consume(ConsumeContext<PaymentSucceeded> context)
    {
        var message = context.Message;

        await _publishEndpoint.Publish(new EmailEvent
        {
            To = "user@example.com",
            Subject = "Payment Received",
            Body = $"Payment for order {message.OrderId} succeeded!"
        });
    }
}
```
### MassTransit will serialize the message and send it to RabbitMQ.

You can have another service or consumer handle that message:

```
public class EmailEventConsumer : IConsumer<EmailEvent>
{
    public Task Consume(ConsumeContext<EmailEvent> context)
    {
        var email = context.Message;
        Console.WriteLine($"[RabbitMQ] Send email to: {email.To}");
        return Task.CompletedTask;
    }
}
```

## 🔐 Durability Comparison

|Feature|`IMediator`|`IPublishEndpoint`|
|---|---|---|
|Transport|In-memory|RabbitMQ (or any broker)|
|Message persistence|❌ No|✅ Yes|
|Retry support|❌ No|✅ Yes (via broker + MassTransit)|
|Dead-lettering|❌ No|✅ Yes|

---

## 🛠 Setup Requirements for MassTransit + RabbitMQ

In `Program.cs`:

```
builder.Services.AddMassTransit(x =>
{
    x.AddConsumers(Assembly.GetExecutingAssembly());
    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host("rabbitmq://localhost");
        cfg.ConfigureEndpoints(context);
    });
});
```
---

## ✅ Best Practice

Use both where appropriate:

- Use `IMediator.Publish` for **in-app workflows** (e.g., update DB → send notification → log).    
- Use `IPublishEndpoint.Publish` to **broadcast events** to other services.
