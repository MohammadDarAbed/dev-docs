# 📘 ASP.NET Core Configuration

ASP.NET Core uses a **Configuration system** to load and manage settings for your application. This system helps you read values from files like `appsettings.json`, environment variables, and other sources in a unified way.

---

## 🔧 What is `IConfiguration`?

- `IConfiguration` is an interface used to **read key-value settings**.
    
- These settings can come from multiple sources like JSON files, environment variables, or command-line arguments.
    
- You can inject it anywhere (e.g., in `Startup`, controllers, services) to read configuration values.
    

---

## 📂 Common Configuration Sources

ASP.NET Core reads configuration values from several sources in a **specific order**. The **last one wins** (i.e., overrides previous values):

1. **Command-Line Arguments**    
2. **Environment Variables**    
3. **User Secrets** (development only)    
4. **appsettings.{Environment}.json** (e.g., `appsettings.Development.json`)    
5. **appsettings.json**    

📝 **Tip**: If a key is defined in multiple places, the one from the source **added later** will override earlier ones.

---

## 📁 Example: appsettings.json

```
{
  "MyKey": "Value123",
  "AppSettings": {
    "SiteName": "My Cool App",
    "Version": "1.0"
  }
}
```
To read these values:
```
string key = Configuration["MyKey"];
string siteName = Configuration["AppSettings:SiteName"];
```

---

## 💻 Reading Configuration in Code

### In `Startup.cs`:
```
public class Startup
{
    public IConfiguration Configuration { get; }

    public Startup(IConfiguration configuration)
    {
        Configuration = configuration;
    }

    public void ConfigureServices(IServiceCollection services)
    {
        string siteName = Configuration["AppSettings:SiteName"];
    }
}
```
### In a Controller:
```
public class HomeController : Controller
{
    private readonly IConfiguration _config;

    public HomeController(IConfiguration config)
    {
        _config = config;
    }

    public IActionResult Index()
    {
        string version = _config["AppSettings:Version"];
        return Content($"Version: {version}");
    }
}
```
---

## 🌍 Using Environment Variables

- Environment variables override `appsettings.json`.    
- Use double underscores `__` for nested keys:    
    - Example: `AppSettings__SiteName=New Name`        
    - Maps to: `Configuration["AppSettings:SiteName"]`        

---

## 🛡️ Using User Secrets (for Development)

- Store sensitive data like API keys securely during development.    
- Run the following in your project directory:    
```
dotnet user-secrets init
dotnet user-secrets set "AppSettings:ApiKey" "my-secret-key"

```
- In `Program.cs`:
```
if (app.Environment.IsDevelopment())
{
    builder.Configuration.AddUserSecrets<Program>();
}
```

## 🧪 Accessing Strongly Typed Settings (Optional)

1. Create a class:
```
public class AppSettings
{
    public string SiteName { get; set; }
    public string Version { get; set; }
}
```
1. Register in `Startup.cs`:
```
services.Configure<AppSettings>(Configuration.GetSection("AppSettings"));
```

1. Inject via `IOptions<AppSettings>`:
  
```
public class HomeController : Controller
{
    private readonly AppSettings _settings;

    public HomeController(IOptions<AppSettings> options)
    {
        _settings = options.Value;
    }

    public IActionResult Index()
    {
        return Content($"Site: {_settings.SiteName}");
    }
}
```
---

## ✅ Best Practices

- 🔐 **Never store secrets** in `appsettings.json`. Use **User Secrets** or environment variables.
    
- 📁 Use `appsettings.{Environment}.json` for environment-specific overrides.
    
- ⚙️ Use `Configuration["Section:Key"]` for nested values.
    
- 📦 Use **strongly-typed options** (`IOptions<T>`) for better maintainability.
    
- 🔄 JSON files support auto-reload by default when changed.
