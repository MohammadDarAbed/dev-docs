# ✅ **What are Action Filters?**
**Action Filters** are a type of **filter** in ASP.NET Core that run **before and/or after controller actions**.
They let you add **extra behavior** to your actions **without changing the action code**.

### 🧠 Think of them like this:

> “Do something **before or after** an action runs.”

---

### ✅ Common uses:

- Logging    
- Validation    
- Authentication
- Caching    
- Modifying the request/response    

---

### ✅ Example:

```
public class MyLogActionFilter : IActionFilter Or IAsyncActionFilter
{
    public void OnActionExecuting(ActionExecutingContext context)
    {
        // Runs before the action
        Console.WriteLine("Action starting");
    }

    public void OnActionExecuted(ActionExecutedContext context)
    {
        // Runs after the action
        Console.WriteLine("Action finished");
    }
}
```

### ✅ How to use:

Apply it on a controller or action:

```
[ServiceFilter(typeof(MyLogActionFilter))]
public IActionResult MyAction()
{
    return Ok();
}
```

### ✅Registration:
there are 3 types of registration:
1. 
int the profram.cs:
this way to register as a global filter.
```
builder.Services.AddController( options => 
{
options.Filters.Add<FilterClass>();
});
```
2. if we want to use the filter for specific controller or method:
```
public class SimpleFilterAttribute : ActionFilterAttribute
{
    public override void OnActionExecuting(ActionExecutingContext context)
    {
        // Do something
    }
}

[SimpleFilter]
public class MyController : ControllerBase
{
    // Actions
}
```

# Filters Vs Middleware:
|Feature|Middleware|Filters|
|---|---|---|
|**Runs Before/After**|Request/response pipeline|Controller/action execution|
|**Access to**|`HttpContext`|`HttpContext`, route data, controller context|
|**Applies To**|All requests (global)|Controllers/actions|
|**Best For**|Cross-cutting concerns (auth, logging, etc.)|MVC-specific logic (model validation, result processing)|
## ✅ أوجه التشابه بين Middleware و Filters

|وجه الشبه|التوضيح|
|---|---|
|✅ **التنفيذ قبل وبعد العمليات**|كلاهما يسمح بتنفيذ منطق "قبل" و"بعد" مرحلة معينة في معالجة الطلب.|
|✅ **دعم الاعتراض أو الإلغاء**|يمكن استخدامهما لإيقاف الطلب من الاستمرار (مثلًا: منع الوصول).|
|✅ **استعمال الـ Dependency Injection**|يمكن حقن الخدمات داخل Middleware أو Filter.|
|✅ **تُستخدم لمعالجة المهام المشتركة (Cross-Cutting Concerns)**|مثل: المصادقة، التسجيل (Logging)، التعامل مع الأخطاء، إلخ.|
|✅ **تنفيذ حسب ترتيب معين**|يُنفذ Middleware حسب الترتيب في `Program.cs`، والفلاتر حسب ترتيب التنفيذ الخاص بها في الـ MVC pipeline.|

## ❌ أوجه الاختلاف بين Middleware و Filters

|وجه الاختلاف|Middleware|Filters|
|---|---|---|
|🔄 **المجال (Scope)**|يعمل على **جميع الطلبات** في التطبيق|يعمل فقط داخل **نظام MVC/Web API** (Controllers & Actions)|
|🛣 **موقع التنفيذ**|يعمل **قبل الوصول للـ Routing و MVC**|يعمل **داخل MVC pipeline** بعد التوجيه|
|🎯 **التخصيص**|لا يمكن تخصيصه بسهولة لمتحكم أو أكشن محدد|يمكن تطبيقه على **أكشن أو كنترولر معين**|
|📦 **أنواع متعددة**|نوع واحد (Middleware)|أنواع متعددة (Authorization, Action, Result, Exception, Resource)|
|🧠 **الوصول للمعلومات**|يصل إلى `HttpContext` فقط|يصل إلى `HttpContext` بالإضافة إلى **الموديل، الأكشن، المعلمات، النتائج**|
|🔄 **التحكم في تدفق الـ MVC**|لا يمكنه التأثير على تنفيذ الأكشن مباشرة|يمكنه تعديل أو منع تنفيذ الأكشن والنتيجة|

---

## 🎯 متى تستخدم Middleware ومتى تستخدم Filters؟

| السيناريو                                      | الأنسب                         |
| ---------------------------------------------- | ------------------------------ |
| تسجيل الدخول أو التحقق من التوكن في كل الطلبات | Middleware                     |
| التحقق من صلاحيات المستخدم في أكشن معين فقط    | Filter (AuthorizationFilter)   |
| التعامل مع الأخطاء العامة                      | Middleware أو Exception Filter |
| تعديل نتيجة الأكشن (مثل تغيير نوع الاستجابة)   | ResultFilter                   |
| التحقق من البيانات قبل تنفيذ الأكشن            | ActionFilter                   |
