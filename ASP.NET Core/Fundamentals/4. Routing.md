# 🛣 Routing in .NET

## 🔹 What is Routing?

- **Routing**: The process of matching incoming **HTTP requests** to **endpoint handlers** in an ASP.NET Core app.    
- It maps **URLs** to **code** (controller actions or minimal API endpoints).
> [!Note] AID
Routing in .NET Core is the mechanism that maps incoming HTTP requests to the appropriate endpoint in the application. It determines how URLs are processed and which code should handle the request, whether it's a controller action in MVC or an endpoint in Minimal APIs.
>
> In ASP.NET Core, routing is **middleware-based**, meaning it’s part of the request pipeline. When a request comes in, the routing middleware parses the URL, matches it against defined route templates, and then invokes the corresponding handler.
> 
>There are two main routing approaches in .NET Core:
>
>1. **Attribute-based routing (used in MVC/Web API)** – where routes are defined using attributes directly on controller actions.
>
>2. **Convention-based or Minimal API routing** – where routes are configured in `Program.cs` using methods like `MapGet`, `MapPost`, etc.
  >
Routing also supports **route parameters, constraints, optional parameters, and query strings**, making it flexible for building RESTful APIs. In addition, ASP.NET Core allows **route grouping**, endpoint naming, and URL generation, which help in organizing and maintaining larger applications.

---

## 🔑 Key Concepts

- **Route Template**: Defines URL patterns.    
    - Example: `/products/{id}`        
- **Parameters**: Dynamic placeholders in routes.    
    - Example: `/users/{userId}/orders/{orderId}`        
- **Constraints**: Restrict valid route parameter values.    
    - Example: `{id:int}` only matches integers.        

---

## 📌 Types of Routing

### 1️⃣ **Controller-based Routing (Attribute Routing)**

- Defined with **attributes** (`[Route]`, `[HttpGet]`, etc.) in controllers.    

Example:

```
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    [HttpGet] // Matches: GET /api/products
    public IActionResult GetProducts() => Ok("All Products");

    [HttpGet("{id:int}")] // Matches: GET /api/products/5
    public IActionResult GetProduct(int id) => Ok($"Product {id}");
}
```

---

### 2️⃣ **Minimal API Routing**

- Define routes directly in `Program.cs` using `MapGet`, `MapPost`, etc.

Example:

```
app.MapGet("/products", () => "All Products");
app.MapGet("/products/{id:int}", (int id) => $"Product {id}");
```

---

## 🧩 Route Templates

### 🔹 Static Routes

```
app.MapGet("/about", () => "About Page");
```

### 🔹 Parameterized Routes

```
app.MapGet("/products/{id}", (int id) => $"Product ID: {id}");
```

### 🔹 Optional Parameters

```
app.MapGet("/hello/{name?}", (string? name) => $"Hello {name ?? "Guest"}");
```

### 🔹 Multiple Parameters

```
app.MapGet("/users/{userId}/orders/{orderId}", (int userId, int orderId) =>
    $"User {userId}, Order {orderId}");
```

---

## 🔒 Route Constraints

### Common Constraints:
- `{id:int}` → Integer only    
- `{name:alpha}` → Alphabet only    
- `{price:decimal}` → Decimal numbers    
- `{date:datetime}` → DateTime value    
- `{length:min(3)}` → Minimum length    

Example:

```
app.MapGet("/products/{id:int:min(1)}", (int id) => $"Product {id}");
```
---

## 🧭 Query vs Route Parameters

### Route Parameter

- Part of the **URL path**.    

```
app.MapGet("/products/{id}", (int id) => $"Product {id}");
```

➡️ Request: `GET /products/5`

### Query Parameter

- Added **after `?` in URL**.    

```
app.MapGet("/search", (string q) => $"Search: {q}");
```

➡️ Request: `GET /search?q=phone`

---

## 🔂 Route Order & Matching

- Routes are matched **in the order they are defined**.    
- More **specific routes** should be defined **before generic ones**.    

Example:

```
app.MapGet("/products/special", () => "Special Route"); app.MapGet("/products/{id}", (int id) => $"Product {id}");
```

✔️ `/products/special` will not be treated as `{id}`.

---

## 🎯 Route Grouping (Minimal API)

Group related endpoints under a **common base route**:

```
var productGroup = app.MapGroup("/products");

productGroup.MapGet("/", () => "All Products");
productGroup.MapGet("/{id:int}", (int id) => $"Product {id}");
productGroup.MapPost("/", (Product p) => $"Added {p.Name}");
```

➡️ Results in:

- `GET /products`    
- `GET /products/5`    
- `POST /products`    

---

## 🔀 Endpoint Naming & URL Generation

- Use `WithName` for named routes (useful for URL generation & redirection):

```
app.MapGet("/products/{id}", (int id) => $"Product {id}")
   .WithName("GetProduct");

app.MapGet("/redirect", (HttpContext ctx) =>
{
    var url = ctx.GetRouteUrl("GetProduct", new { id = 10 });
    return Results.Redirect(url!);
});
```

## 🛡 Middleware vs Routing

- **Middleware** runs for every request.    
- **Routing** determines which endpoint handles a request.    

The request pipeline:

`HTTP Request → Middleware → Routing → Endpoint → Response`

**Middleware** is a component in the ASP.NET Core request pipeline that processes HTTP requests and responses. It can perform tasks like authentication, logging, exception handling, or modifying requests before they reach the endpoint. Middleware is executed sequentially for every request.

**Routing**, on the other hand, is a specific middleware responsible for matching incoming requests to their corresponding endpoints (like controller actions or minimal API handlers). Once routing identifies the endpoint, the request is dispatched to it.

---

### **Key Difference:**

|**Aspect**|**Middleware**|**Routing**|
|---|---|---|
|**Purpose**|Processes every request and response|Maps requests to endpoints (controller/API)|
|**Execution**|Runs sequentially in the pipeline|Runs after middleware pipeline reaches routing|
|**Scope**|Can modify or short-circuit the request|Focuses only on endpoint selection|
|**Examples**|Authentication, Logging, Exception Handling|Matching `/products/5` to `GetProduct(int id)`|

---

## ✅ Best Practices

- Use **clear, resource-based routes** (`/products`, `/orders`).    
- Keep route **parameters simple** and meaningful.    
- Apply **constraints** to avoid invalid requests.    
- Group endpoints logically using **`MapGroup`**.    
- For large apps, prefer **Controller routing** for better organization.
