## 📌 APIs in .NET

### 🔹 What is an API?

- **API (Application Programming Interface)**: A contract that allows two applications to communicate with each other.    
- In .NET, APIs are commonly **HTTP-based (RESTful APIs)** built using **ASP.NET Core**.    

---

### 🏗 Types of APIs in .NET

1. **Controller-based API (Traditional)**    
    - Uses **Controllers** that inherit from `ControllerBase`.        
    - Organizes endpoints in separate controller classes.        
    - Example:

```
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    [HttpGet]
    public IActionResult GetProducts() => Ok(new[] { "Product1", "Product2" });
}
```

2. **Minimal API (New in .NET 6+)**    
    - Lightweight way of creating APIs without controllers.        
    - Define endpoints directly in `Program.cs`.        
    - Ideal for **microservices** or **small APIs**.        

---

### 🔑 REST API Principles

- **Stateless**: Each request is independent.    
- **Resource-based**: Exposes resources (e.g., `/products`).    
- **CRUD Operations**:    
    - `GET`: Retrieve data.        
    - `POST`: Create new resource.        
    - `PUT/PATCH`: Update existing resource.        
    - `DELETE`: Remove resource.        

---

## ⚡ Minimal APIs

### 🔹 What is a Minimal API?

- Introduced in **.NET 6**.    
- **Definition**: A simplified API programming model for small, fast HTTP APIs with minimal boilerplate code.    
- **Goal**: Reduce ceremony (no controllers, no attributes, fewer files).    

---

### ✨ Features of Minimal APIs

✅ No Controllers → Endpoints are defined inline.  
✅ Lightweight & Fast → Lower overhead than MVC.  
✅ Perfect for **microservices, serverless APIs, or prototyping**.  
✅ Supports **dependency injection**, **filters**, **middleware**, and **OpenAPI (Swagger)**.

---

### 🛠 Minimal API Example

```
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// GET endpoint
app.MapGet("/hello", () => "Hello, World!");

// POST endpoint with input
app.MapPost("/greet", (User user) => $"Hello {user.Name}");

app.Run();

record User(string Name);
```

---

### 🔗 Binding Data in Minimal APIs

- Automatically binds from:    
    - **Query strings**        
    - **Route parameters**        
    - **Request body (JSON)**        
    - **Form data**        

Example:

```
app.MapGet("/products/{id}", (int id) => $"Product ID: {id}");
app.MapPost("/products", (Product p) => $"Added: {p.Name}");
```

---

### 🧩 Dependency Injection in Minimal APIs

```
builder.Services.AddScoped<IProductService, ProductService>();

app.MapGet("/products", (IProductService service) =>
{
    return service.GetAllProducts();
});
```

---

### 🛡 Authentication & Authorization in Minimal APIs


```
app.MapGet("/secure", [Authorize] () => "Secure Data");
```

---

### 📦 Adding Swagger for Minimal APIs

```
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

app.UseSwagger();
app.UseSwaggerUI();
```

---

### 🆚 Minimal API vs Controller API

|Feature|Minimal API|Controller API|
|---|---|---|
|Boilerplate|Very low|Higher (Controllers, Attributes)|
|Complexity|Simple projects|Large/Complex projects|
|Flexibility|Less (inline)|More (filters, MVC features)|
|Performance|Faster (less overhead)|Slightly slower|

---

### 🔥 When to Use Minimal APIs?

✅ Small microservices  
✅ Prototypes or MVPs  
✅ Serverless functions  
✅ Simple CRUD APIs

❌ For complex apps → Prefer **Controller-based APIs**.

---

## 🚀 Best Practices for APIs (Both Controller & Minimal)

- ✅ Use **DTOs (Data Transfer Objects)** to avoid exposing entities.    
- ✅ Validate inputs with `FluentValidation` or `DataAnnotations`.    
- ✅ Enable **Swagger/OpenAPI** for documentation.    
- ✅ Handle **global error handling** with middleware.    
- ✅ Use proper **HTTP status codes** (`200 OK`, `400 BadRequest`, `404 NotFound`).

### 📌 Overview

**gRPC** is a high-performance, open-source, universal RPC framework developed by Google. It uses **Protocol Buffers (Protobuf)** for serialization and **HTTP/2** as its transport protocol.

In contrast, a **traditional API** (like REST) typically uses **JSON over HTTP/1.1** and follows resource-based communication patterns.

---

### 🧠 Key Concepts

#### **1. RPC (Remote Procedure Call)**

- Instead of sending requests to "resources" (like in REST), you call **methods** directly on a remote server as if they were local functions.    
- gRPC generates client & server code from a `.proto` definition file.    

#### **2. Protocol Buffers (Protobuf)**

- gRPC’s interface definition language (IDL).    
- Binary, strongly typed, compact → **faster and smaller** than JSON (80% faster).    

#### **3. HTTP/2 Transport**
- Supports **multiplexing**, **streaming**, **header compression**, and **bi-directional communication**.    

---

### 🆚 gRPC vs REST API

|Feature|**gRPC**|**REST API**|
|---|---|---|
|**Transport Protocol**|HTTP/2|HTTP/1.1 (or HTTP/2 in some cases)|
|**Data Format**|Protocol Buffers (binary)|JSON / XML (text-based)|
|**Performance**|High (smaller payload, faster serialization)|Moderate (larger payload, slower parsing)|
|**Streaming**|Native client, server, and bidirectional streaming|Not native, usually via WebSockets|
|**Typing**|Strongly typed via `.proto`|Weakly typed (JSON)|
|**Tooling**|Auto code generation for multiple languages|Manual model creation in most cases|
|**Caching**|Limited built-in HTTP caching|Strong native HTTP caching support|
|**Readability**|Harder to debug manually (binary)|Human-readable (JSON)|
|**Best Use Cases**|High-performance, low-latency, internal microservice communication|Public APIs, browser-based clients, simple CRUD APIs|

---

### 🔍 How gRPC Works

![[grpc.png]]
#### 1. Define the service

You create a `.proto` file:

```
syntax = "proto3";

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest {
  string name = 1;
}

message HelloReply {
  string message = 1;
}
```
#### 2. Generate code

Run the `protoc` compiler to generate:
- Server interface (to implement the logic)    
- Client stubs (to call the service)    

#### 3. Implement & run
Server implements the service methods.  
Client calls them as if they were local functions.

---

### 📡 Communication Types in gRPC

1. **Unary RPC**    
    - One request → One response (like a normal function call).        
2. **Server Streaming**    
    - One request → Multiple responses streamed back.        
3. **Client Streaming**    
    - Multiple requests streamed → One response.        
4. **Bidirectional Streaming**    
    - Both client and server stream messages simultaneously.        

---

### 🏗 Advantages of gRPC

- **Performance**: Smaller, faster payloads due to Protobuf + HTTP/2.    
- **Strong typing**: Compile-time safety.    
- **Streaming**: First-class support for bi-directional streaming.    
- **Code generation**: Reduces boilerplate code.    
- **Cross-language support**: Works across multiple languages seamlessly.    

---

### ⚠️ Limitations of gRPC

- **Not human-readable**: Harder to debug without tools like `grpcurl`.    
- **Browser support**: Native gRPC is not directly supported in browsers (needs gRPC-Web).    
- **Caching**: Less built-in caching compared to REST.    
- **Learning curve**: Requires learning Protobuf and gRPC concepts.    

---

### 🗂 When to Use gRPC vs REST

✅ **Use gRPC when:**
- You need **low-latency** communication.    
- You’re working with **microservices** internally.    
- You need **streaming** (real-time updates, telemetry, etc.).    
- You want **multi-language support** without writing separate APIs.    

🚫 **Use REST when:**
- You’re building **public APIs** consumed by browsers or third-party clients.    
- Human readability/debugging is important.    
- HTTP caching and wide tooling support are required.    
- Simplicity is preferred over raw performance.    

---

### 📖 Example: gRPC vs REST Request

**REST (JSON)**
```
POST /hello
Content-Type: application/json

{
  "name": "Alice"
}
```

Response:

```
{
  "message": "Hello Alice"
}
```

**gRPC (Protobuf)**

- Client sends a binary-encoded `HelloRequest` over HTTP/2.    
- Server responds with a binary-encoded `HelloReply`.    
- No need to write manual serialization/deserialization code.
### 📌 HTTP/1.1 vs HTTP/2 — and How Envoy Proxy Enables HTTP/2 for gRPC
### 🔹 Overview

- **HTTP/1.1** has been the standard for web communication since 1999.
    
- **HTTP/2** was introduced in 2015 to address performance bottlenecks in HTTP/1.1, enabling faster, more efficient communication.
    
- **gRPC** _requires_ HTTP/2 because it needs multiplexing, binary framing, and streaming support.
    
- The **problem**: browsers and some environments can’t always speak raw HTTP/2 with gRPC → This is where **Envoy Proxy** comes in.
    

---

## 🆚 HTTP/1.1 vs HTTP/2

|Feature|**HTTP/1.1**|**HTTP/2**|
|---|---|---|
|**Connections**|One request per TCP connection (or pipelining with head-of-line blocking)|Multiple concurrent requests on a single connection (multiplexing)|
|**Data Format**|Text-based (headers and body in plain text)|Binary framing (faster to parse)|
|**Header Compression**|None (or minimal)|HPACK compression reduces header size|
|**Streaming**|Limited support|Full duplex streaming supported|
|**Performance**|Slower for many small requests|Faster, fewer connections, reduced latency|
|**TLS Requirement**|Optional|Optional in spec, but almost always used in browsers|

---

## ⚠️ Why gRPC Needs HTTP/2

- **gRPC uses streaming** — sending/receiving multiple messages in the same connection.    
- **Multiplexing** lets multiple gRPC calls happen in parallel over a single TCP connection.    
- **Binary framing** matches well with Protocol Buffers (Protobuf) for efficiency.    

If you try to run gRPC over HTTP/1.1:
- No multiplexing → slower requests.    
- No native bidirectional streaming → some gRPC features break.    

---

## 🛑 The Problem: Browsers and Raw HTTP/2 gRPC

- Browsers **can** use HTTP/2 for normal HTTPS requests, but:    
    1. They **don’t allow raw HTTP/2 frames** to be handled directly in JavaScript.        
    2. gRPC’s binary framing and streaming **aren’t supported natively** in `fetch` or `XMLHttpRequest`.        
- This means a browser can’t directly talk to a gRPC server using HTTP/2.    

---

## 🛠 How Envoy Proxy Fixes It

### 1. Acts as a **Protocol Translator**
- Browser sends **gRPC-Web** requests using HTTP/1.1 or HTTP/2 (in a compatible form).    
- Envoy **translates** gRPC-Web into standard gRPC over HTTP/2 for the backend.    

```
Browser (gRPC-Web over HTTP/1.1) → Envoy Proxy → Backend (gRPC over HTTP/2)
```

---

### 2. Enables HTTP/2 Between Services

- Even if the client is limited to HTTP/1.1, Envoy speaks **full HTTP/2** with backend gRPC services.   
- This ensures:    
    - Multiplexing        
    - Streaming        
    - Binary framing support        

---

### 3. Extra Benefits

- **Load balancing** between multiple gRPC servers.    
- **TLS termination** — Envoy handles HTTPS and talks plain HTTP/2 internally.    
- **Service discovery** — no need to hardcode backend IPs.    
- **Observability** — metrics, tracing, logging.

## 📍 Summary

- **HTTP/1.1**: Simple, text-based, one-request-at-a-time (or limited pipelining).    
- **HTTP/2**: Binary, multiplexed, supports streaming → essential for gRPC.    
- **Problem**: Browsers can’t do raw gRPC over HTTP/2.    
- **Solution**: Envoy Proxy translates gRPC-Web to gRPC-over-HTTP/2 for backend services and unlocks HTTP/2 features in distributed systems.
