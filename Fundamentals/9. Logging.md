# 📘 ASP.NET Core Logging 
---

## ✅ What is Logging?

**Logging** is the process of recording events or information about how your application behaves during execution. It’s like the app telling you what it’s doing.

---

## 🎯 Why Do We Use Logging?

Logging is **essential** in every application for:

- 🐞 **Debugging**: Find and fix issues during development.    
- 📊 **Monitoring**: Understand what’s happening in production.    
- 🚨 **Error Tracking**: Capture and analyze failures.    
- 🔒 **Auditing**: Log important business or security actions.    
- 📈 **Performance Analysis**: Log response times and bottlenecks.    

---

## 👤 Who Uses Logging?

- **Developers**: To debug and understand app behavior.    
- **QA/Testers**: To verify and reproduce bugs.    
- **DevOps / IT Admins**: To monitor app health in production.    
- **Security Teams**: To detect suspicious behavior.    

---

## 🧱 Logging Levels (Types)

ASP.NET Core has different **log levels** to classify the importance of each message:

|Level|When to Use|
|---|---|
|🔍 Trace|Very detailed information for deep debugging.|
|🐛 Debug|Info useful during development/debugging.|
|ℹ️ Info|Normal app events (e.g., "User logged in").|
|⚠️ Warning|Something unexpected, but app still works.|
|❌ Error|Something failed (e.g., exception caught).|
|💥 Critical|Serious failure, app might crash or stop.|
|🚫 None|Disable logging completely.|

---

## 🧰 How to Use Logging in ASP.NET Core

ASP.NET Core uses the `ILogger<T>` interface to log messages.

### 🔧 Step 1: Inject `ILogger<T>`
```
public class HomeController : Controller
{
    private readonly ILogger<HomeController> _logger;

    public HomeController(ILogger<HomeController> logger)
    {
        _logger = logger;
    }

    public IActionResult Index()
    {
        _logger.LogInformation("Index page accessed at {Time}", DateTime.UtcNow);
        _logger.LogWarning("This is a warning example.");
        _logger.LogError("An error happened!");

        return View();
    }
}

```

## ⚙️ Logging Configuration in `appsettings.json`

```
.{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft": "Warning",
      "Microsoft.Hosting.Lifetime": "Information"
    }
  }
}
```

- **Default**: Controls your app logs.    
- **Microsoft**: Controls logs from the ASP.NET Core framework.    
- You can override this in `appsettings.Development.json` or `appsettings.Production.json`.   

---

## 🧪 Environment-Based Logging

You can have different logging levels for each environment:

- **Development**:
 
 ```
 {
  "Logging": {
    "LogLevel": {
      "Default": "Debug",
      "Microsoft": "Information"
    }
  }
}
```
- **Production**:

```
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft": "Error"
    }
  }
}
```

## 🖥️ Built-in Logging Providers

ASP.NET Core supports multiple output **providers**:

|Provider|Output Destination|
|---|---|
|Console|Terminal or Command Prompt|
|Debug|Visual Studio Output window|
|EventLog|Windows Event Viewer|
|TraceSource|Older tracing system|
|Azure App Insights / Seq / Serilog|External systems|

To configure them:

```
builder.Logging.ClearProviders(); // Optional
builder.Logging.AddConsole();
builder.Logging.AddDebug();
```
---

## 🧠 Best Practices

- ✅ Use **LogInformation** for regular activity.    
- ⚠️ Use **LogWarning** for expected but unusual cases.    
- ❌ Use **LogError** and **LogCritical** for exceptions or crashes.    
- 🔒 **Never log sensitive data** like passwords or tokens.    
- 💬 Use **structured logging**:

```
_logger.LogInformation("User {UserId} logged in at {Time}", userId, DateTime.UtcNow);
```
