## 🧩 Liskov Substitution Principle (LSP)

### 📖 Definition

> **Objects of a superclass should be replaceable with objects of its subclasses without breaking the application.**

This means:

- Subclasses must behave in a way that does **not violate the expectations** set by the base class.   
- A derived class should only extend behavior, **not change it in an incompatible way**.    
> For example if you have class B Inherits from class A then class A should be replacable by class B without any changes:

Class A{}
Class B extends A{}
Class C extends A{}
::::::::::
A objectA = new B();
A objectB = new C();

---

### 🚩 **Why LSP?**

- Prevents inheritance misuse where a subclass breaks the logic of its parent.    
- Ensures polymorphism works correctly.    
- Avoids unexpected bugs when substituting a subclass where the base class is expected.
- 
### 🛑 **Violation of LSP: Rectangle vs Square**

```
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }

    public int Area() => Width * Height;
}

public class Square : Rectangle
{
    public override int Width
    {
        set { base.Width = base.Height = value; }
    }

    public override int Height
    {
        set { base.Width = base.Height = value; }
    }
}
```
**Problem:**

- A `Square` is forced to inherit from `Rectangle`.    
- Changing width also changes height, breaking `Rectangle`'s expected behavior.    

Test case:

```
Rectangle rect = new Square();
rect.Width = 5;
rect.Height = 10;
Console.WriteLine(rect.Area()); // Expected 50, but gets 100
```

This violates **LSP** because substituting `Square` for `Rectangle` changes expected behavior.

---

### ✅ **Fixing the LSP Violation**

We separate `Rectangle` and `Square` into independent classes that both implement a shared abstraction (`IShape`).

```
public interface IShape
{
    int Area();
}

public class Rectangle : IShape
{
    public int Width { get; set; }
    public int Height { get; set; }
    public int Area() => Width * Height;
}

public class Square : IShape
{
    public int Side { get; set; }
    public int Area() => Side * Side;
}
```

Now:

- Both `Rectangle` and `Square` follow **LSP** because they don’t interfere with each other's behavior.    
- They can both be used polymorphically via `IShape`.

### 🛑 **Violation of LSP (Bird, Sparrow, Ostrich)**

```
public class Bird
{
    public virtual void Fly()
    {
        Console.WriteLine("I am flying!");
    }
}

public class Sparrow : Bird
{
    public override void Fly()
    {
        Console.WriteLine("Sparrow is flying!");
    }
}

public class Ostrich : Bird
{
    public override void Fly()
    {
        throw new NotImplementedException("Ostrich cannot fly!");
    }
}
```
**Problem:**

- `Ostrich` inherits from `Bird` but cannot fly.    
- Code expecting a `Bird` that can fly will break if it receives an `Ostrich`.

Example:
```
Bird bird = new Ostrich();
bird.Fly(); // ❌ Throws exception! Violates LSP.
```

### ✅ **Fixing LSP Violation**

We split the abstraction into **`Bird`** and **`FlyingBird`** to separate behaviors.

```
public abstract class Bird
{
    public abstract void Eat();
}

public abstract class FlyingBird : Bird
{
    public abstract void Fly();
}

public class Sparrow : FlyingBird
{
    public override void Eat()
    {
        Console.WriteLine("Sparrow is eating seeds.");
    }

    public override void Fly()
    {
        Console.WriteLine("Sparrow is flying.");
    }
}

public class Ostrich : Bird
{
    public override void Eat()
    {
        Console.WriteLine("Ostrich is eating plants.");
    }
}
```
Now:

- **`Sparrow`** inherits from `FlyingBird` (can fly).    
- **`Ostrich`** inherits from `Bird` (no flying behavior).    
- Code that requires a flying bird will only accept `FlyingBird`.    

Example:
```
FlyingBird sparrow = new Sparrow();
sparrow.Fly(); // ✅ Works

Bird ostrich = new Ostrich();
ostrich.Eat(); // ✅ Works (no Fly method available)
```
---

### 💡 Key Takeaways

- **LSP ensures subclasses don't alter parent behavior in unexpected ways.*    
- Avoid **over-constraining subclasses** with forced inheritance (favor interfaces).    
- A subclass should **only extend, never contradict** base class behavior.    

---

### 🔑 Real-World Examples

- **Collections:** A `ReadOnlyCollection` should not allow adding/removing items if used as a general `Collection`.    
- **Geometry:** Avoid forcing "Square is a Rectangle" if behavior contradicts.    
- **Bank Accounts:** A `SavingsAccount` shouldn't allow overdraft if substituted where a `CurrentAccount` is expected.
